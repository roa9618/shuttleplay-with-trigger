# 베타 운영 체크리스트

이 문서는 셔틀플레이를 실제 배드민턴 모임에서 테스트하기 전, 당일, 종료 후에 확인할 내용을 정리합니다.

베타 테스트의 목적은 기능이 돌아가는지만 보는 것이 아닙니다. 실제 사람이 있는 체육관에서 출석, 매칭, 코트 운영, 결과 입력, 실시간 동기화가 끊기지 않는지 확인해야 합니다.

## 베타 테스트 기준

| 항목 | 기준 |
| --- | --- |
| 예상 참가자 | 20~30명 |
| 예상 코트 수 | 3~4코트 |
| 운영자 기기 | 태블릿 또는 노트북 |
| 참가자 기기 | 개인 스마트폰 |
| 표시 장비 | 큰 화면 디스플레이 또는 노트북 화면 |
| 핵심 검증 | 출석, 자동 매칭, 경기 운영, 결과 기록, 실시간 동기화 |

## 테스트 목표

1. 참가자가 QR 또는 링크로 세션에 접속할 수 있는지 확인합니다.
2. 출석 체크와 참가자 상태 관리가 현장에서 자연스럽게 동작하는지 확인합니다.
3. 자동 매칭 결과가 운영자와 참가자 모두에게 납득 가능한지 확인합니다.
4. 경기 후보 큐가 실제 코트 운영 흐름과 맞는지 확인합니다.
5. 경기 시작, 종료, 결과 입력 흐름이 끊기지 않는지 확인합니다.
6. 운영자 화면, 참가자 화면, 큰 화면 경기판이 실시간으로 동기화되는지 확인합니다.
7. MMR과 개인 기록이 과도하거나 이상하게 반영되지 않는지 확인합니다.
8. 참가자가 화면 흐름을 어렵지 않게 이해하는지 확인합니다.
9. 현장에서 생긴 예외 상황을 기록하고 개선점을 남깁니다.

- 참가자 상태 변경

## 당일 시작 전 확인

- 체육관 네트워크 상태를 확인합니다.
- 운영자 기기가 인터넷에 연결되어 있는지 확인합니다.
- 참가자 스마트폰에서 접속 테스트를 합니다.
- 서버와 DB가 정상 동작 중인지 확인합니다.
- WebSocket 연결이 정상인지 확인합니다.
- QR 코드가 정상 표시되는지 확인합니다.
- 큰 화면 경기판이 준비되어 있는지 확인합니다.
- 운영자 화면에 세션 정보, 출석 현황, 참가자 목록이 표시되는지 확인합니다.

## 현장 테스트 순서

### 1. 세션 입장

- 참가자가 QR 또는 링크로 접속합니다.
- 회원은 로그인 후 참여합니다.
- 비회원은 이름, 성별, 나이대, 급수를 입력하고 참여합니다.
- 운영자는 참가자 목록에 새 참가자가 표시되는지 확인합니다.

확인할 것:

- QR 접속 실패가 없는지
- 모바일 화면이 너무 복잡하지 않은지
- 비회원 등록이 빠르게 끝나는지
- 중복 이름이 있을 때 구분할 수 있는지

### 2. 출석

- 참가자는 도착 후 출석 체크를 합니다.
- 운영자는 출석 현황을 확인합니다.
- 지각 예정자와 불참자를 등록합니다.

확인할 것:

- 출석 후 참가자 상태가 매칭 가능한 상태로 바뀌는지
- 지각 예정자는 자동 매칭에서 제외되는지
- 불참자는 자동 매칭에서 제외되는지
- 운영자 화면과 참가자 화면이 같은 상태를 보여주는지

### 3. 자동 매칭

- 운영자가 경기 유형과 플레이 스타일을 선택합니다.
- 자동 매칭을 생성합니다.
- 경기 후보 큐를 확인합니다.

확인할 것:

- 경기 가능한 참가자만 후보에 들어갔는지
- 경기 중이거나 다음 경기 예정인 참가자가 중복 포함되지 않았는지
- 2연속 휴식자가 우선 배정되는지
- 3연속 경기가 방지되는지
- 파트너 중복이 줄어드는지
- 팀 MMR 차이가 지나치게 크지 않은지
- 매칭 설명이 운영자가 이해할 수 있는지

### 4. 경기 시작과 종료

- 운영자가 후보 경기를 코트에 올립니다.
- 참가자 화면에 다음 경기 또는 경기 시작 정보가 표시됩니다.
- 큰 화면 경기판에 현재 경기 정보가 반영됩니다.
- 경기가 끝나면 운영자가 종료 처리합니다.

확인할 것:

- 코트 번호가 정확한지
- 참가자 상태가 `PLAYING`으로 바뀌는지
- 경기 종료 후 참가자 상태가 다시 대기 상태로 바뀌는지
- 경기판이 지연 없이 바뀌는지

### 5. 경기 결과 입력

- 운영자가 승리 팀을 선택합니다.
- 가능하면 점수까지 입력합니다.
- MMR 변화와 개인 기록 반영을 확인합니다.

확인할 것:

- 승패만 입력해도 저장되는지
- 점수 입력 시 점수 차 보정이 적용되는지
- MMR 변화가 과도하지 않은지
- 결과 입력 후 리포트와 개인 기록이 갱신되는지

### 6. 예외 상황

현장에서 아래 상황을 일부러 만들어 확인합니다.

- 참가자가 늦게 도착함
- 참가자가 중도 퇴장함
- 경기 결과를 잘못 입력함
- 후보를 생성한 뒤 특정 참가자를 제외해야 함
- 코트 수가 줄어듦
- 네트워크가 잠시 끊김
- 참가자 화면을 새로고침함

확인할 것:

- 운영자가 손으로 복구할 수 있는지
- WebSocket이 끊긴 뒤 REST API로 최신 상태를 다시 맞추는지
- 잘못된 결과 수정 후 MMR과 기록이 다시 계산되는지

## 종료 후 확인

- 세션을 종료합니다.
- 세션 리포트를 확인합니다.
- 결과 미입력 경기가 남아 있는지 확인합니다.
- 참가자별 경기 수 편차를 확인합니다.
- 오래 쉰 참가자가 있었는지 확인합니다.
- 같은 파트너/상대 반복이 과했는지 확인합니다.
- MMR 변화가 납득 가능한지 확인합니다.
- 운영자가 불편했던 흐름을 기록합니다.
- 참가자 피드백을 정리합니다.

## 베타 피드백 기록 양식

```txt
일시:
장소:
참가자 수:
코트 수:
운영자 기기:
큰 화면 사용 여부:

잘 동작한 부분:

불편했던 부분:

버그:

매칭이 이상했던 사례:

MMR이 과하게 변한 사례:

다음 테스트 전 수정할 것:
```

## 운영 중 우선 대응할 문제

즉시 대응:

- 운영자 화면 접속 불가
- 출석 체크 불가
- 자동 매칭 생성 불가
- 경기 시작/종료 불가
- 결과 입력 불가
- WebSocket 전체 동기화 실패

테스트 후 수정:

- 매칭 품질이 낮은 사례
- 특정 참가자 경기 수 쏠림
- 파트너/상대 반복
- 모바일 화면 사용성 문제
- 알림이 늦거나 눈에 잘 안 띄는 문제

베타 이후 검토:

- PWA 푸시 알림 안정화
- 운영자 메모
- 월별 기록
- 매칭 가중치 자동 조정
- 운영자 피드백 기반 매칭 품질 개선
