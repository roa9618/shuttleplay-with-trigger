# 요구사항

이 문서는 베타 운영을 기준으로 반드시 필요한 요구사항을 정리합니다. 세부 구현은 기능을 만들면서 조정할 수 있지만, 아래 흐름과 우선순위는 프로젝트의 기본 기준으로 둡니다.

## 우선순위

| 우선순위 | 의미 |
| --- | --- |
| P0 | 베타 운영에 반드시 필요한 기능 |
| P1 | 베타 완성도와 사용성을 높이는 기능 |
| P2 | 베타 이후 고도화해도 되는 기능 |

## 사용자와 참여

### 회원가입 `P1`

회원은 이름, 로그인 정보, 성별, 나이대, 급수를 입력해 계정을 만들 수 있어야 합니다. 회원가입 시 급수에 따라 복식 MMR과 혼복 MMR 초기값을 부여합니다.

필수 입력값:

- 이름
- 이메일 또는 아이디
- 비밀번호
- 성별
- 나이대
- 급수

처리 기준:

- 필수값이 빠지면 가입할 수 없습니다.
- 이미 사용 중인 로그인 식별자는 사용할 수 없습니다.
- 회원은 개인 기록, MMR, 월별 운동 기록을 저장할 수 있습니다.

### 로그인 `P1`

회원은 로그인 후 개인 기록과 모임 정보를 볼 수 있어야 합니다. 운영자 권한이 있는 사용자는 모임 관리 기능에 접근할 수 있어야 합니다.

### 비회원 참여 `P0`

비회원도 QR 또는 링크로 세션에 들어와 당일 운동에 참여할 수 있어야 합니다.

필수 입력값:

- 이름
- 성별
- 나이대
- 급수

처리 기준:

- 비회원은 특정 세션의 참가자로 등록됩니다.
- 비회원은 당일 경기 현황 확인과 출석 체크가 가능합니다.
- 비회원의 장기 기록 저장은 제한합니다.
- 동일 세션 안에서 표시 이름이 겹치면 운영자 확인 또는 식별값 추가가 필요할 수 있습니다.

## 모임과 세션

### 모임 생성 `P0`

운영자는 반복적으로 사용할 수 있는 고정 모임을 만들 수 있어야 합니다.

입력값:

- 모임명
- 활동 장소
- 기본 코트 수
- 기본 운동 시간
- 모임 설명

처리 기준:

- 모임 생성자는 기본 운영자가 됩니다.
- 모임 정보는 세션 생성 시 기본값으로 사용할 수 있습니다.

### 모임 정보 수정 `P1`

운영자는 모임명, 장소, 기본 코트 수, 기본 운동 시간, 설명을 수정할 수 있어야 합니다. 수정된 기본값은 이후 생성되는 세션부터 반영합니다.

### 모임 멤버 관리 `P1`

운영자는 고정 멤버를 추가, 수정, 제외할 수 있어야 합니다. 기존 멤버는 새 세션에 빠르게 추가할 수 있어야 합니다.

### 운동 세션 생성 `P0`

운영자는 특정 날짜의 운동 세션을 만들 수 있어야 합니다.

입력값:

- 모임 ID
- 운동 날짜
- 시작 시간
- 종료 예정 시간
- 코트 수
- 세션 유형

처리 기준:

- 세션 생성 후 초대 링크와 QR을 만들 수 있어야 합니다.
- 세션에는 참가자, 출석, 경기 후보 큐, 실제 경기, 결과 기록이 연결됩니다.
- 코트 수는 1 이상이어야 합니다.

### 세션 종료 `P0`

운영자는 운동이 끝난 뒤 세션을 종료할 수 있어야 합니다.

처리 기준:

- 진행 중인 경기가 있으면 종료 전 확인이 필요합니다.
- 세션 종료 후 추가 매칭 생성은 제한합니다.
- 세션 종료 시 요약 리포트를 생성합니다.

## 초대 링크와 QR

### 초대 링크 생성 `P0`

운영자는 특정 세션에 연결된 초대 링크를 만들 수 있어야 합니다. 사용자는 이 링크로 세션 참여 화면에 접근합니다.

### QR 코드 생성 `P0`

운영자는 초대 링크를 QR로 보여줄 수 있어야 합니다. 참가자는 QR을 스캔해 참가 등록 또는 출석 체크를 진행합니다.

## 출석과 참가자 상태

### 출석 체크 `P0`

참가자는 세션에 도착한 뒤 출석 체크를 할 수 있어야 합니다.

처리 기준:

- 출석 체크한 참가자는 출석 완료 상태가 됩니다.
- 운영자 정책에 따라 경기 가능 상태로 전환됩니다.
- 출석하지 않은 참가자는 자동 매칭 후보에 포함하지 않습니다.

### 지각 예정 등록 `P1`

참가자는 도착 예정 시간과 사유를 등록할 수 있어야 합니다. 지각 예정자는 도착 전까지 자동 매칭 후보에서 제외합니다.

### 불참 처리 `P1`

참가자 또는 운영자는 불참 상태를 등록할 수 있어야 합니다. 불참자는 자동 매칭 후보에서 제외합니다.

### 참가자 상태 관리 `P0`

| 상태 | 설명 | 자동 매칭 포함 |
| --- | --- | --- |
| `NOT_ARRIVED` | 아직 도착하지 않음 | 제외 |
| `LATE_EXPECTED` | 지각 예정 | 제외 |
| `ATTENDED` | 도착했지만 경기 가능 전환 전 | 조건부 |
| `AVAILABLE` | 경기 가능 | 포함 |
| `WAITING` | 경기 가능하지만 현재 미배정 | 포함 |
| `SCHEDULED` | 다음 경기 후보에 포함됨 | 제외 |
| `PLAYING` | 현재 경기 중 | 제외 |
| `RESTING` | 운영자가 휴식 처리 | 제외 |
| `LEFT` | 중도 퇴장 | 제외 |
| `ABSENT` | 불참 | 제외 |

자동 매칭은 기본적으로 `AVAILABLE`, `WAITING` 참가자를 기준으로 생성합니다.

## 자동 매칭

### 자동 매칭 생성 `P0`

시스템은 경기 가능한 참가자를 기준으로 최적의 경기 후보를 만들어야 합니다.

계산에 사용하는 값:

- 성별
- 급수
- 나이대
- 복식 MMR
- 혼복 MMR
- 경기 유형
- 오늘 경기 수
- 연속 경기 수
- 연속 휴식 수
- 최근 파트너 이력
- 최근 상대 이력
- 신규 참가자 여부

처리 기준:

- 가장 높은 품질 점수를 가진 경기 후보를 우선 큐에 등록합니다.
- 코트 수만큼 후보를 선택하되, 참가자가 여러 후보에 중복 포함되지 않도록 합니다.
- 조건을 완전히 만족하지 못해도 가능한 최선의 후보를 만들고, 필요한 경우 운영자에게 낮은 품질 안내를 보여줍니다.

### 경기 유형 선택 `P0`

운영자는 자동 매칭 시 경기 유형을 선택할 수 있어야 합니다.

| 유형 | 기준 |
| --- | --- |
| `MENS_DOUBLES` | 남자 복식 우선 |
| `WOMENS_DOUBLES` | 여자 복식 우선 |
| `MIXED_DOUBLES` | 남녀 2명씩, 팀별 남녀 1명씩 우선 |
| `ANY` | 성별보다 실력 균형과 참여 기회 우선 |

경기 유형은 절대 조건이 아니라 우선 조건입니다. 인원 구성이 맞지 않으면 유연 매칭을 적용할 수 있고, 이 경우 운영자에게 이유를 보여줍니다.

### 플레이 스타일 선택 `P0`

| 스타일 | 우선 기준 |
| --- | --- |
| `FUN` | 다양한 조합, 경기 횟수 균등, 중복 최소화, 신규 참가자 배려 |
| `COMPETITIVE` | 팀 평균 MMR 차이 최소화, 실력 균형, 경기 몰입도 |

### 연속 경기 제한 `P0`

- 기본 정책은 1경기 후 1회 휴식입니다.
- 2연속 경기는 상황에 따라 허용합니다.
- 3연속 경기는 강하게 방지합니다.

### 연속 휴식 방지 `P0`

- 1회 휴식은 허용합니다.
- 2연속 휴식 참가자는 다음 경기 우선 배정 대상입니다.

### 파트너와 상대 중복 최소화 `P0/P1`

- 파트너 중복은 `P0`으로 강하게 줄입니다.
- 상대 중복은 `P1`로 관리하되, 가능한 한 반복을 줄입니다.

### 고정/회피 조합 `P1`

운영자가 설정한 고정 조합 또는 회피 조합을 매칭 점수에 반영합니다. 다만 절대 금지 조건으로 두면 매칭 자체가 막힐 수 있으므로 기본은 강한 가중치로 처리합니다.

### 신규 참가자 배려 `P1`

신규 참가자는 첫 경기까지 너무 오래 기다리지 않도록 하고, 지나치게 강한 경기에 바로 배정하지 않습니다.

### 매칭 결과 설명 `P1`

운영자는 자동 매칭 결과의 주요 선택 사유를 볼 수 있어야 합니다.

예시:

- 2연속 휴식 참가자를 우선 배정했습니다.
- 팀 전력 차이가 낮은 조합입니다.
- 파트너 중복이 없는 조합입니다.
- 혼복 인원 부족으로 유연 매칭이 적용되었습니다.

## 경기 후보 큐

### 경기 후보 큐 생성 `P0`

자동 매칭 결과는 실제 경기로 바로 시작하지 않고 경기 후보 큐에 등록합니다. 코트가 비면 큐에서 우선순위가 높은 경기를 올립니다.

### 일부 후보 재생성 `P1`

운영자는 특정 경기 후보를 유지한 상태로 나머지 후보만 재생성할 수 있어야 합니다.

## 경기 운영

### 경기 시작 `P0`

운영자는 경기 후보를 실제 경기로 시작 처리할 수 있어야 합니다.

처리 기준:

- 경기 참가자는 `PLAYING` 상태가 됩니다.
- 해당 경기는 진행 중 경기 목록에 표시됩니다.
- 운영자 화면, 참가자 화면, 큰 화면 경기판에 WebSocket으로 반영됩니다.

### 경기 종료 `P0`

운영자는 진행 중인 경기를 종료 처리할 수 있어야 합니다.

처리 기준:

- 경기 상태는 종료로 변경됩니다.
- 참가자 상태는 경기 가능 또는 대기 상태로 전환됩니다.
- 경기 결과 입력 대기 상태가 됩니다.

### 경기 결과 입력 `P0`

운영자는 경기의 승패와 점수를 입력할 수 있어야 합니다.

처리 기준:

- 승패만 입력해도 MMR 계산이 가능해야 합니다.
- 점수까지 입력되면 점수 차 기반 보정값을 적용합니다.
- 결과 입력 후 개인 기록과 세션 리포트를 갱신합니다.

### 경기 결과 수정 `P1`

잘못 입력된 경기 결과는 수정할 수 있어야 합니다. 수정 시 기존 MMR 반영을 확인하고, 수정된 결과 기준으로 개인 기록과 MMR을 다시 계산합니다.

## 기록과 리포트

### MMR 관리 `P0/P1`

- 복식 MMR과 혼복 MMR은 분리합니다.
- 급수에 따라 초기 MMR, 하한선, 소프트 캡을 적용합니다.
- 점수 차가 있으면 MMR 변동폭을 보정합니다.
- 경기 결과 수정 시 MMR 이력을 남기고 다시 계산할 수 있어야 합니다.

### 개인 운동 기록 `P1`

회원은 일자별 경기 수, 승패, 승률, 점수, MMR 변화, 운동 시간을 확인할 수 있어야 합니다.

### 월별 운동 기록 `P2`

회원은 월간 출석 횟수, 경기 수, 승패, MMR 변화, 승률을 확인할 수 있어야 합니다.

### 세션 종료 리포트 `P1`

운영자는 세션 종료 후 총 참가자 수, 총 경기 수, 평균 경기 수, 결과 미입력 경기, 참가자별 기록을 확인할 수 있어야 합니다.

## 화면

### 운영자 대시보드 `P0`

운영자는 한 화면에서 세션 정보, 출석 현황, 참가자 목록, 현재 경기, 경기 후보 큐, 결과 입력 대기 경기, 참가자별 경기 수를 볼 수 있어야 합니다.

### 참가자 현황 화면 `P0`

참가자는 개인 기기에서 본인의 현재 상태, 다음 경기 예정 여부, 파트너, 상대, 코트 번호, 오늘 경기 수를 확인할 수 있어야 합니다.

### 큰 화면 경기판 `P1`

체육관 큰 화면에는 현재 진행 중인 경기, 다음 경기 후보, 코트별 상태, 경기 종료 대기 정보가 표시되어야 합니다.

## 알림

### 웹 내부 알림 `P0`

다음 경기 배정, 경기 시작, 코트 이동 같은 필수 안내는 웹 내부 알림으로 제공해야 합니다. 토스트, 배너, 모달, 효과음, 진동을 상황에 맞게 사용합니다.

### PWA 푸시 알림 `P2`

PWA 푸시는 기기와 브라우저 제한이 있으므로 보조 알림으로 둡니다. 필수 안내는 웹 내부 알림을 기준으로 보장합니다.
